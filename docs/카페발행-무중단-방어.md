# 카페 발행 무중단 — '중간에 끊김/누락' 방어 총정리 (sub2 이식용)

> **어제(2026-07-20) 발행이 중간에 멈춰 예약분이 누락된 사고**를 막으려고 넣은 방어를 한곳에 모았다.
> 대부분 **이미 코드에 반영돼 있어 git 으로 받으면 sub2 도 동일**하다. 이 문서는 "무엇을·왜"를 남겨 재발을 막는다.
> OS/전원 레벨(절전·시작프로그램·로그인유지)은 [PC 상시가동 설정](PC-상시가동-설정.md) 참고 — 이 문서는 **코드/런타임 레벨**.

---

## 0. 어제 무슨 일이었나 (재발 방지 대상)

**증상**: 밤 12시까지 예약해둔 카페 발행이 하나도 안 나갔다. 리스너는 떠 있는 것처럼 보였다.

**원인**: 발행 중 스마트에디터가 띄운 대화상자(beforeunload/alert)를 처리하는 이벤트 콜백 `_on_dialog` 안에서,
대화상자가 우리가 처리하기 전에 스스로 닫히면 `"No dialog is showing"` 예외가 터졌다.
이 예외가 **이벤트 핸들러 안에서 발생**해 리스너 프로세스를 통째로 죽였고, 이후 예약 발행이 전부 누락됐다.

**교훈**: 무인 발행에서 "한 번의 예외 = 전체 정지 = 조용한 누락"이 가장 위험하다. 아래 방어는 이걸 겹겹이 막는다.

## 1. 코드 레벨 방어 (전부 git 반영 — sub2 가 받으면 동일)

| # | 방어 | 위치 | 막는 것 |
|---|---|---|---|
| 1 | **대화상자 크래시 삼킴** | `publish_cafe.py` `_on_dialog` try/except | 어제 그 크래시. 콜백 예외로 프로세스가 죽지 않게 |
| 2 | **일시오류=실패 대신 대기 유지** | 커밋 `954ea20` | 로그인만료·크롬꺼짐·일시오류 시 글을 fail 시키지 않고 **pending 유지 → 복구 후 자동 재개** |
| 3 | **리스너 자가복구 + .bat 재시작** | 커밋 `2956d8d`, `run_cafe_listener.bat` `:loop` | 파이썬이 어떤 이유로 죽어도 **30초 뒤 자동 재기동** |
| 4 | **재시작해도 발행 간격 유지** | `publish_listener.py` `_init_last_pub_from_db` | 재시작 직후 몰아 발행하지 않게(DB `done_at` 기준 복원) |
| 5 | **좀비 회수(reaper)** | `_reap_stale` (processing 30분 초과 → pending) | 죽은 프로세스가 잡고 있던 행이 영영 안 나가는 것 |
| 6 | **등록됐는데 미확정 경고** | `_warn_stuck_posted` | posted 인 채 방치된 행 감지(사람 확인용, 자동 재발행은 안 함) |
| 7 | **중복 발행 방지(CAS)** | 조건부 PATCH `status=eq.pending` | 두 리스너/재시작이 같은 글을 두 번 올리는 것 |
| 8 | **이미지 파일창 재시도** | 커밋 `b77e1cc` | 사진 선택창이 포커스/타이밍으로 안 뜰 때 발행 실패 |
| 9 | **에디터 비우기·팝업정리·preflight** | 커밋 `7644527` | 이전 글 잔재로 본문 오염·빈 글 발행 |
| 10 | **정지 시간대** | `CAFE_STOP_AT=HH:MM` | 지정 시각 후 발행 중단(세션 유지 핑만) |

> 핵심 원칙: **한 건의 실패가 전체를 멈추지 않는다.** 실패는 그 글만 fail/pending 으로 격리되고 루프는 계속된다.

## 2. OS/전원 레벨 (→ PC-상시가동-설정.md)

- **절전·최대절전 끄기** (절전 들어가면 크롬·파이썬 정지 → 최우선)
- **시작프로그램 VBS** (재부팅 후 자동 복구, 담당 PC 한 대만)
- **네이버 "로그인 상태 유지" 체크** (크롬 죽어도 세션 유지)
- **`.bat`/`.vbs` 는 CRLF** (LF 면 자동시작이 조용히 실패 — `.gitattributes` 가 강제)

## 3. 더티클리닉 스크립트 발행의 무중단 (리스너와 별개)

더티클리닉은 큐/리스너가 아니라 **스크립트 루프**(`auto_publish_dirty.py`)로 돈다. 여기에도 같은 원칙 적용:

| 방어 | 내용 |
|---|---|
| 각 건 독립 | 지역별 `try/except` — 한 건 실패해도 다음 지역 계속 |
| 배너 3회 재시도 | AI 배너 생성 실패(429·네트워크) 시 재시도 |
| **완료 기록** | 성공 지역을 `published_regions.json` 기록 → **재시작 시 건너뛰고 이어감**(중복·누락 방지) |
| 실패 기록 | `failed_regions.json` → 나중 재시도 |
| **watchdog `.bat`** | 프로세스가 죽으면 재시작 → 완료분 스킵하고 남은 지역부터. 어제 같은 정지를 자동 복구 |

상세: [더티클리닉 자동발행](더티클리닉-자동발행.md)

## 4. sub2 이식 절차

1. `git fetch && git checkout -B sub2 origin/main` — **위 코드 방어(§1)가 전부 딸려온다**(별도 작업 없음)
2. `docs/PC-상시가동-설정.md` 대로 OS/전원 설정(절전 끄기·시작프로그램·로그인유지)
3. `.env`(로컬): `CAFE_STOP_AT`, `CAFE_KEEPALIVE_MIN`, `CAFE_REAP_MIN`, `CAFE_MAX_ATTEMPTS` 등은 기본값으로 충분
4. Supabase: `docs/cafe-publish-queue.sql` 의 `attempts`/`claimed_at` 컬럼(없으면 리스너가 claim 단계에서 실패)
5. **리스너/발행은 한 대에서만** — 두 대 동시 = 중복 발행([[multi-computer-git-workflow]])

## 5. 무중단 점검 체크리스트

- [ ] `run_cafe_listener.bat` 의 `:loop` 자동재시작이 살아있나(파이썬 죽여도 30초 뒤 부활하는지)
- [ ] `_on_dialog` try/except 가 코드에 있나(어제 크래시 방어의 핵심 — grep `No dialog`)
- [ ] `CAFE_STOP_AT` 이 원하는 시각인가
- [ ] 절전 꺼져 있나(`powercfg /q SCHEME_CURRENT SUB_SLEEP STANDBYIDLE` → 0x0)
- [ ] (더티클리닉) `published_regions.json` 로 재시작 시 이어가는지 / watchdog `.bat` 이 크래시만 재시작하는지
- [ ] 발행 담당 PC가 **한 대뿐**인지(중복 방지)
